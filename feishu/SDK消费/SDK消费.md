# SDK消费
## 需求背景
SDK消费的设计目标从产品的层面来说，是为了弥补Kafka消费存在着Kafka的消息格式和日志服务的日志格式不一致的问题，因为格式不一致就一定会有格式的转换，下面简单描述下日志服务的格式和Kafka消息格式的区别
1. 首先讲一下Kafka原生的流程：kafka是一个上传io里面有多个record，消费的时候一次最小消费一个io，一般通过maxfetchBytes的这个参数拉多个IO，一般来说每个IO都是压缩后数据（消费的时候不会指定压缩格式，因为每个IO存的时候会带压缩格式）Kafka拉数据是直接把IO拉出来然后给客户端，客户端来解压缩和解码
2. 基于第一点，为了兼容kafka的client所以kafka的proxy层一定要做一个消息格式的转换，这个转换的流程是，先把日志服务的io数据解压缩，然后再解码，然后还有一个步骤，因为日志服务的数据分两种，一种叫做单行全文，一种叫做kv，一般来说都是kv，所以要把这个kv数据转换成json数据，所以还涉及到json的序列化，json的序列化是最耗性能的大概耗时占40%，然后还要再encode成kafka的record，然后还要再压缩（其实这个过程我们做了很多性能优化）
   1. kafka默认一个fetch请求可以拉多个分区的数据，但是一般来说性能最优的场景就是一个fetch请求只拉一个分区的数据，这个分区数据的大小默认是1MB，我们服务端的审计日志表明，一般来说一个IO，的最大的一个平均是20到30KB，最大的账号的IO有600KB左右，基于这个配置的话其实其实能吃到我们服务端的所有多核优化的（我们服务端的规格是32C），
   2. 我们服务端的多核优化是：因为一次会拉取超过20个IO，那么每个IO的解压缩和反序列化（这里的反序列化指的是从日志服务专有的格式反序列化（pb格式））都是要并行，然后并行出来之后把所有的日志放在一个list里面，然后每1000条日志去做分block，然后再每1000条日志作为一个block做json序列化和kafka record encode，这样的性能对比完全的单核串行，同样大小的数据时间延迟能变成原来的3/1
   3. 还有一个json序列化的优化：其实最开始是想直接替换Json库来性能优化，结果发现针对50个KV以上的Json序列化，主流的GO里面的库差不多性能差距都在10%之内，后面想到一个法子是因为我们日志的格式都是String，就是KV都是String（这个格式是用过户上传的时候就定了的，所以是一定的），所以这个json的序列化优化也是走了很多弯路的，最开始的时候的原始方法是我们的日志对象转一个map，然后用json序列化这个map，然后换了几个json的库找了个最快的，然后性能还是不行，因为我们对外的性能规格是10MB/S，然后后面的优化思路是，打火焰图，看cpu占用，发现一个是json序列化占用多，这个是一直问题，一个是小对象gc，这个应该是因为我一条日志就是一个map，map里面kv都是string，所以很多小对象，然后第二版就是把这个map对象复用起来，用了一个go里面的对象池的机制，减少了GC空间换时间，内存占用上升了百分之3（因为原本就很低，基本没有占用内存）然后速度快了30%，差不多有8MB/S了，但是还是不够，然后还有一个因为我们的日志是KV，然后k和v其实都是string，而且一定只有一层不会递归（这个很关键），然后我去看了一些json库的代码，发现有两个可以优化的点
      1. 就算我序列化map，这个map里面的value是需要递归的，因为他为了保证api的兼容性，一般都是map[object]objet这种（当然这个在go里面是interface）所以他会去递归判断value是不是一个对象，然后递归序列化这个对象，这一点我们不用，因为我们产品层面就规定了我们的kv只有一层，你的value无论如何都是一个raw字符串，所以我不需要做这个递归的逻辑
      2. 那些json库都需要去判断k和v的类型，计算我传一个map[string]string过去，他还是要判断，这个倒不用判断了，因为我们kv一定是string
   4. 基于上面两点，解法就很简单了 我直接for循环append byte数组就好了，这样还能少一个map（这个主要是会影响GC），类似于java的stringbuilder，这样一搞差不多性能快了一倍，10MB的IO 200多MS返回，kafka原生是20MS多一个数量级，但是达到了设计要求
3. 上面说的这些都是说明一个问题，就是为啥日志服务走kafka消费的性能比原生的慢一个数量级（其实一般来说别的无状态的请求可以通过增加并发来解决吞吐量的问题，但是kafka其实是一个流式消费，就是我一个分区一定是要被连续消费的，客户端无法通过多个客户端消费一个分区来提升消费吞吐量，所以个人理解时延是消息队列消费的最重要指标，那从这个指标上来讲，日志服务的kafka比起原生kafka性能差上一个数量级）
## 需求设计目标
1. 性能上（消费时延上）对齐kafka（由于kafka是存算一体的，都放在一个broker上，我们是把数据放在shard server，会多一跳，整体会比kafka多点点延迟）
2. 支持http api的接口直接消费（相当于客户端管理消费进度）
3. 支持消费组消费，可以少量消费重复
## 设计
### 日志存储的时候构建cursor索引
cursor的设计和kafka的offset不一样，kafka是连续且单调递增的Int64，cursor这个东西其实本质上是《文件唯一ID+文件偏移量》，kafka和rocketMQ其实是一条消息一个offset，这个offset是中间抽象出来的一层，它后面通过一个叫offsetIndex的索引来映射了offset和消息在文件中的位置，这个逻辑相对而言比较复杂，我们的设计是把文件位置信息直接存到cursor里面，那么用户通过cursor来拉数据的时候不用进过offset到文件位置的查找，其实速度上差不多，因为耗时大头不在这里，主要是方案上要更清爽一点，kafka优点太复杂了
这两个方案的对比是这样的
1. kafka是可以支持只消费一条数据的（因为它的逻辑是通过这个offset，去它构建的一个offsetIndex去二分的找最接近的一个io，然后把这个iodecode出来找到对应的数据，所以如果kafka一直去拉io之间的offset，性能会低一点，一般来说正常的消费都是客户端来拉一个io，然后消费完这个io再消费另一个IO，这样都是按照io fetch数据的，但是kafka 提交的offset不是按照io来提交的，可能提交的offset是一个io中间的offset，那么这个时候，加入客户端重启了，要通过服务端返回的offset去拉数据，那么这个第一次的数据的fetch时延会比后面的都要高点，因为kafka的broker里面有解压，decode，然后再封装成新的io包的逻辑，但是这个只是说在消费者重启的时候才有，一般来说我们用一个消息队列流式消费都是启动了就不动了，一直消费下去），我们这种套路是一次至少消费一个IO，因为用户提交到服务端的cursor是IO粒度的，区别就是你客户端还没有提交消费进度就重启了的时候这样客户端会重复消费的数据量不一样，kafka的会少点这个是kafka的优势，但是这个其实这个来说kafka也不能做到完全不重复
2.kafka还有一个优势，就是offset是单调递增且连续的，方便算消费堆积，消费堆积是消息队列很重要的一个监控指标，kafka就可以用提交的消费进度和当前这个分区的最大的offset做一个相减，就能拿到消费lag，但是我们的这个设计不行，我们只能提供大小堆积，比如堆积了多少MB（因为我cursor里面有便宜量），和时间堆积（就是对比最新的数据和当前消费的数据的时间差，当然这个kafka也能做）各有利弊，不过我理解还是kafka的更好一点，因为数据条数更直观
刚才说的是cursor的格式，然后就是sdk的消费逻辑其实和kafka一样，也是消费组启动的时候先去服务端查询消费组的消费进度，如果这个消费进度没有，就通过客户端设置的一个initOffset（这个可以代表最老的offset，最新的offet，或者某个时间戳）通过这个时间戳来查询curosr，所以为了支持这个功能就是需要在我们每个io写入的时候去构建一个cursorIndex，这个里面是一个稀疏索引，就是存的cursor和时间戳的关系，然后查找的时候二分查找，找到最近的就去按照这个cursor去读io，一个个读，然后读出来IO数据，对比IO上存的那个时间戳，找到就返回这个IO的cursor，因为我们构建这个索引的时候是按照大小和时间窗口聚合的，就是多少时间窗口or多少大小IO写一个索引的entry，所以这个读IO的大小是可控的
### http直接走api消费（类似于拉模型）
通过cursor直接去拉去对应的io，但是由于是走http的接口，要给用户返回可读的数据，所以需要再服务度对io包进行解压缩然后序列化成json给用户，这个地方有开销，但是对比kafka少了转换成kafka消息的一步，所以性能介于原生kafka和日志服务kafka之间
### 消费组消费
1. 服务端做分区的分配，分配的视图存在etcd里面
2. 客户端上报的心跳也存在etcd里面（我们的pod是无状态的，用户请求我们api的任何一个pod过来，api的pod去etcd里面这个消费组的living_nodes这个路径下面putkey，就是put自己的memberID（这个memeberID就是一个客户端生成的UUID，其实本质上我们想要的就是分布式的情况下的一个不重复的ID），续心跳）
3. 每个消费组都有一个类似于kafka的coordiantor的节点，这个是选了一个leader来做视图分配决定的，就是有个leader会告诉每个coordiantor节点，你这个节点需要处理哪些group的消费视图分配，然后这个coordiantor就会监听这个给他分配的消费组的Living_nodes路径，然后通过去DB里面查询我们的Topic元数据拿分区信息，然后做一个分区的分配，策略其实很简单，就是roundrobin，分配好了之后通过心跳返回给消费者，告诉他你要消费哪些分区（这个其实是优点抄rocketMQ的，不过rocketMQ是客户端分配，所以他其实是通过心跳的请求，把自己的memeberID上报到broker，然后把broker当作一个广播，广播给其他消费者，因为每个消费者都会和他要消费的topic的所有分区leader维持心跳，不过我们做的其实类似是一个无状态的，主要是我们内部通过etcd在所有的api pod里面同步了视图分配了，会更清爽，但是我们服务端的瓶颈会更大）
4. 因为我们的pod是无状态的所以我们通过心跳返回消费分区的时候我会去etcd里面查询视图分配的结果，然后返回给client
5. 为了避免消费重复我们还做了一个事情，就是每次用户fetch请求的时候都会带上他的memberID，就是其实这个就是会告诉客户端，我现在哪个消费者来消费我的哪个分区，这个可以这个分区已经被切换到另外一个消费者了，但是以前的消费者还来消费，可以避免重复消费，如果服务端发现这个消费者来消费的分区不是服务端分配的，会报错，客户端会重新发送心跳请求来更新消费的分区信息
6. 3里面说的瓶颈就是我们为了把我们的pod做成无状态的，监听了大量的etcd的key，就比如说我集群里面有30个pod然后，然后这30个pod都需要全部监听所有消费组的分配视图，（这个就是一个笛卡尔积，我扩我的pod，我etcd的监听数就会线性增长）这样是不行的，所以我们做了一个改造，就是首先把服务端的那个http的keep alive打开，默认是5S不用这个链接就关闭，但是因为我们其实是拉模型，默认是1s来拉一次数据，所以这个链接大概率是可以复用的，然后只有我发了fetch请求的pod，里面根据收到的fetch请求的消费组ID来监听这个消费组的视图分配，并且有个后台任务去扫，多久没有请求的消费组的etcd watcher会被close掉，这样差别做到了线上50个pod的情况下，etcd的连接数只放大来3倍，并且不会随着pod的数量的增长水平上涨
